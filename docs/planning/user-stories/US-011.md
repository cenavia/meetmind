## User Story

**Como** Marina o cualquier usuario  
**Quiero** recibir mensajes claros cuando la información es incompleta o hay errores **y**, cuando exista transcripción, poder **copiar el texto final** con facilidad  
**Para** entender qué pasó, qué puedo hacer, distinguir el análisis útil de las advertencias del sistema y **reutilizar la transcripción** en otro documento o herramienta

---

## Descripción ampliada

### Comportamiento del sistema (backend y lógica)

El sistema debe manejar de forma elegante los casos límite y los errores:

- **Información incompleta:** texto muy corto, sin participantes identificables, transcripción de baja calidad.
- **Errores de procesamiento:** fallos en transcripción, timeout del modelo, archivo corrupto o no soportado.
- **Estados parciales:** resultados parciales (por ejemplo solo temas, sin acciones).

El usuario **nunca** debe ver errores técnicos crudos (stack traces, trazas, mensajes de excepción internos). En su lugar: mensajes en lenguaje natural, estado `partial` o `failed` con descripción legible, y uso de `processing_errors` para advertencias acumulables (coherente con el modelo de US-012).

### Presentación en la interfaz (definiciones de diseño)

La experiencia debe alinearse con la UI acordada para MeetMind:

1. **Separación visual del diagnóstico y del resultado**  
   Las advertencias y errores (`processing_errors`, descripciones de fallo) se muestran en un **bloque dedicado** (p. ej. callout o panel claramente titulado: “Avisos” / “No se completó todo”), **separado** del Markdown o secciones del análisis (participantes, temas, acciones, minuta, resumen). Así el usuario no confunde contenido generado con mensajes del sistema.

2. **Estados legibles, no solo color**  
   Los estados `completed`, `partial` y `failed` se comunican con **texto explícito** (etiqueta o insignia con palabra legible), complementando el color para accesibilidad.

3. **Microcopy y contexto**  
   - Mensajes que indiquen **acción sugerida** cuando aplique (reintentar, usar otro archivo, pegar más texto, comprobar conexión).  
   - En validación previa al envío (texto vacío, archivo no válido), mensajes **junto al área de entrada** o en el mismo flujo que el botón principal, no solo en consola o API.

4. **Contenido técnico fuera del camino principal**  
   Detalles dirigidos a quien despliega el servicio (p. ej. backend de transcripción NUBE/LOCAL, variables de entorno) permanecen en **secciones colapsables** (p. ej. acordeón “Detalles técnicos”), no mezclados con mensajes de error orientados al usuario final.

5. **Coherencia entre vistas**  
   Cuando exista **historial de reuniones** (navegación lateral o pestaña aparte), el detalle de una reunión con error o estado parcial debe usar el **mismo patrón**: bloque de avisos + estado textual + secciones de resultado (vacías o parciales según corresponda).

6. **Estados vacíos**  
   Antes de procesar, la zona de resultado puede mostrar un texto de **estado vacío** amable; los errores de “no enviaste nada” no deben sustituir de forma confusa ese estado cuando el usuario aún no ha actuado.

7. **Transcripción final copiable**  
   Cuando el flujo haya producido una **transcripción completa** (p. ej. a partir de audio o video), la interfaz debe mostrar ese texto en un control que permita **copiarlo al portapapeles** de un gesto (p. ej. botón “Copiar” nativo del componente en Gradio, como en `Textbox` con `buttons=["copy"]` / API equivalente según versión, u otro patrón accesible). El usuario debe poder pegar el contenido en un correo, documento o bloc de notas sin seleccionar manualmente un bloque largo. Si no hubo paso de transcripción (solo texto pegado o archivo ya textual), este bloque puede ocultarse o mostrarse vacío según el diseño de pantalla.

### Layout moderno (Gradio + criterios visuales)

Referencia explícita a **frontend-design** (dirección clara, tipografía, color, espacio, motion, detalle) y a la **API de layout de Gradio**, afinada a MeetMind como **herramienta de productividad** (no demo genérico ni estética “plantilla IA”).

#### Dirección visual

- **Propósito y tono:** interfaz **refinada y orientada a tareas** (legible, calmada, jerárquica), con personalidad propia del producto: suficientemente cuidada para no parecer el tema por defecto sin criterio, sin competir con el contenido (minutas, avisos). Evitar el cliché de gradientes violetas sobre blanco y combinaciones tipográficas genéricas sin decisión.
- **Diferenciación:** coherencia entre **shell** (navegación + marco), **zona de trabajo** (entrada + acciones) y **zona de lectura** (resultados, transcripción, avisos), de modo que un usuario recuerde “dónde miro” en cada fase del flujo.

#### Shell y navegación (Gradio)

- **`gr.Blocks(theme=..., css=...)`** como contenedor raíz: aplicar un **`gr.themes` base** (p. ej. `Soft`, `Base` o builder) y ajustar **tokens** (`set()`, `builder()`) para **primarios, fondos, radios y bordes** alineados a la paleta elegida (un color de acento dominante + neutros; acentos para estados `completed` / `partial` / `failed` coherentes con el tema, no solo rojo/verde sueltos).
- **`gr.Sidebar`** (colapsable, `position` y `width` acordes al contenido): navegación entre **“Procesar reunión”** e **“Historial”** (cuando exista), o equivalente con **radio/botones** dentro del sidebar. El área principal queda reservada al flujo activo.
- Opcional: **`gr.Navbar`** si en la versión de Gradio disponible encaja mejor con el producto; en caso contrario, sidebar + título en cabecera del `Column` principal.

#### Composición espacial (área principal)

- **`gr.Row` + dos `gr.Column`** con `scale` definido: columna **entrada** (tabs o radio para texto vs archivo, archivo, botones, panel de estado/HTML del loader) y columna **salida** (avisos, estado, transcripción, análisis). En viewports estrechos Gradio apila; el orden debe seguir siendo lógico (entrada antes que resultado o según prueba en móvil).
- **`gr.Tabs`** dentro de la columna de entrada para **modo texto / modo archivo** (alternativa al bloqueo mutuo actual), reduciendo fricción cognitiva.
- **`gr.Accordion`** para: (1) análisis estructurado por secciones (participantes, temas, acciones, minuta, resumen); (2) **“Detalles técnicos”** (NUBE/LOCAL, env); (3) transcripción larga si debe quedar plegable.
- **`gr.Group`** opcional para agrupar visualmente “Acciones” (Procesar, Limpiar) y separarlos del contenido editable.

#### Tipografía y color

- Ajustar tema Gradio para **jerarquía clara** (títulos vs cuerpo vs `info` de componentes). Si el tema lo permite, cargar **fuente distintiva** vía `theme` o **`css` adicional** (p. ej. una familia para titulares y otra para cuerpo), evitando la sensación “solo sistema” cuando el proyecto decida invertir en marca.
- Reutilizar **variables CSS** del tema (`var(--body-text-color)`, `--border-color-primary`, etc.) en estilos custom del loader y de **callouts de avisos** para **modo claro/oscuro** sin duplicar lógica.

#### Motion y microinteracciones

- Mantener o refinar la **barra de progreso animada** del loader como **momento de impacto único** (coherente con frontend-design: mejor una animación bien hecha que muchas distracciones).
- Estados de error/éxito: transiciones **CSS suaves** en opacidad o borde de los callouts si se implementan vía `gr.HTML` + CSS inyectado (sin abusar).

#### Componentes Gradio relevantes

- **`Textbox`** para transcripción de solo lectura con **copiar** (`buttons`).
- **`Markdown`** para resultados enriquecidos; valorar `buttons=['copy']` donde aplique la versión.
- **`HTML`** para panel de estado/avisos con HTML + **CSS scoped** al prefijo de clase del proyecto (como el loader actual).
- Guías: [Controlling Layout](https://www.gradio.app/guides/controlling-layout), [Themes](https://www.gradio.app/guides/themes), [Custom CSS and JS](https://www.gradio.app/guides/custom-CSS-and-JS), [Sidebar (docs)](https://www.gradio.app/docs/gradio/sidebar).

## Criterios de Aceptación

### Lógica y API

- [ ] Mensajes claros cuando no se identifican participantes (p. ej. valor vacío o texto tipo “No identificados”, sin error técnico).
- [ ] Advertencia cuando el texto es muy corto o de calidad insuficiente, reflejada en `processing_errors` cuando aplique.
- [ ] Estado `partial` cuando hay resultados incompletos pero utilizables.
- [ ] Estado `failed` con descripción legible cuando falla el procesamiento.
- [ ] Campo `processing_errors` disponible para advertencias o errores no fatales.
- [ ] No se muestran errores técnicos crudos al usuario final (ni en API consumida por la UI en formato expuesto en pantalla).

### Interfaz (Gradio)

- [ ] `processing_errors` y descripciones de fallo se muestran en un **bloque visual distinto** del contenido del análisis (participantes, temas, etc.).
- [ ] El estado de la reunión (`completed` / `partial` / `failed`) es **identificable por texto**, no solo por color.
- [ ] Mensajes de validación locales (sin texto/archivo, archivo no válido) son **comprensibles** y aparecen en el flujo de la pantalla de procesamiento.
- [ ] La UI no mezcla en el mismo párrafo mensajes de usuario y referencias a configuración interna del servidor; lo segundo va a **contenido colapsable** si se muestra.
- [ ] Tras implementar historial, el detalle de ítems `failed` o `partial` replica el mismo esquema de avisos + resultado.
- [ ] Si la API o el stream exponen el **texto de transcripción final**, la UI lo presenta en un componente con **acción explícita de copiar** al portapapeles (o equivalente accesible).
- [ ] El área de transcripción copiable está **claramente etiquetada** (p. ej. “Transcripción”) y no se confunde con el bloque de avisos ni con el resumen ejecutivo.
- [ ] La app usa **`gr.Blocks` con `theme=`** (no solo valores por defecto sin criterio); colores de acento y neutros son **coherentes** entre botones, bordes y paneles custom (loader, avisos).
- [ ] **Shell moderno:** `gr.Sidebar` (o patrón equivalente documentado) separa **procesamiento** e **historial** cuando el historial exista; la zona principal usa **dos columnas** con `scale` para entrada vs salida en escritorio.
- [ ] Entrada por **tabs o radio** para texto vs archivo (según diseño cerrado en implementación), acorde a reducir confusión frente a campos mutuamente excluyentes.
- [ ] Resultado y/o transcripción organizados con **`Accordion`** o **`Tabs`** donde mejore el escaneo; detalles técnicos en acordeón **cerrado por defecto**.
- [ ] Estilos adicionales (**`css=`** en `Blocks`) usan **variables del tema** donde sea posible para soportar apariencia clara/oscura sin romper contraste de avisos y estados.

## Estados y mensajes

| Estado | Descripción para usuario | En UI |
|--------|--------------------------|--------|
| completed | Procesamiento exitoso | Etiqueta de éxito + resultado completo en secciones habituales |
| partial | Algunos resultados; revisar avisos | Etiqueta “Parcial” o equivalente + bloque de avisos visible + secciones con lo disponible |
| failed | Error; revisar descripción | Etiqueta “Error” o equivalente + mensaje en bloque de avisos; sin stack trace |

## Estrategia de robustez

1. **Validación en API:** texto vacío, archivo no soportado, tamaño excesivo.
2. **Validación en preprocess:** longitud mínima de texto, señales de calidad.
3. **Try/except en nodos:** capturar errores, registrar en servidor, devolver mensaje amigable y estado coherente.
4. **Post-procesamiento:** detectar resultados vacíos y etiquetar como `partial` si hay otros datos útiles.
5. **Capa UI:** mapear siempre respuesta estructurada (estado + `processing_errors` + cuerpo) al patrón visual acordado (avisos separados, texto de estado accesible) dentro del **layout de dos columnas** y del **shell** con sidebar cuando aplique.
6. **Transcripción:** si el contrato API/SSE incluye el texto transcrito, la UI lo enlaza al componente con copiar; si aún no existe el campo, coordinar con la historia o tarea que exponga `transcript` (o nombre acordado) antes de cerrar este criterio.
7. **Tema y layout:** centralizar decisiones de `theme` y CSS compartido para que nuevos componentes (avisos, badges de estado) no diverjan visualmente del resto de la app.

## Notas técnicas

- Mapeo explícito de excepciones técnicas a mensajes de usuario y a códigos/estados de negocio.
- La API debe permitir a la UI construir la vista sin parsear mensajes ad hoc; preferir campos estructurados (`status`, `processing_errors`, mensajes de error de dominio).
- Detalles de despliegue (Whisper en nube/local, claves) no forman parte del mensaje de error principal hacia Marina.
- En Gradio, revisar compatibilidad de versión para botón **copiar** en `Textbox` / `Markdown` (`buttons`) y para **`gr.Sidebar`** (versión mínima del paquete).
- Verificar que la versión instalada soporta los componentes de layout elegidos (`Sidebar`, `Navbar`, builder de themes); si no, documentar alternativa (tabs superiores + columnas).
- **Frontend-design (adaptación):** la app es herramienta B2B/B2E; la “dirección atrevida” del skill se traduce en **composición espacial clara, tipografía y tema deliberados**, no en decoración que reste legibilidad a minutas y errores.

## Personas vinculadas
